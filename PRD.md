# AI 错题本 PRD

## 1. 目标

AI 错题本帮助学生把错题从“记录下来”变成“分析错因、安排复习、最终掌握”。

本次 MVP 不做完整错题管理系统，而是优先验证：

> AI 错因归纳 + 智能复习队列，能否把错题从“死档案”转化为可行动的学习任务。

## 2. 背景与问题分析

当前错题本的核心问题不是“没有错题可复习”，而是错题记录后没有形成有效学习闭环。

问题链路可以概括为：

> 错题很多 -> 分类粗糙 -> 不知道错因 -> 不复习或复习低效 -> 留存低、二次练习率低

具体来看：

第一，错题分类维度过粗。现有错题通常按科目、时间简单堆叠，缺少对“错因”和“知识点关联”的深层识别，导致学生无法从大量错题中发现共性问题。

第二，用户留存和二次练习率偏低。错题本 7 日留存低于 10%，二次练习率仅 5%，说明学生虽然会记录错题，但没有持续复习动力。

第三，现有推荐机制偏旧。推荐更多是推原题或随机推相似题，缺少基于学生掌握度的动态判断。

因此，本次方案的重点不是增加错题数量，而是提升错题的结构化归因能力和复习转化效率。

## 3. 现状数据分析

### 3.1 用户规模

当前 DAU 约 10 万，说明错题本已经是一个高频学习场景，而不是低频工具。

付费学生占比约 60%，意味着该功能触达大量高价值用户。错题复习体验的改善，可能进一步影响付费用户留存和续费意愿。

同时，月活跃错题数人均 45 条，说明用户并不缺少错题数据。真正的问题不是“没有错题可复习”，而是错题积累后缺少结构化归因和复习规划。

所以这里的产品取舍是：

> 本次 MVP 不优先解决错题录入量问题，而是优先解决已有错题如何被 AI 自动归因、排序并转化为复习任务。

### 3.2 用户分群

当前用户可以分为三类：

- “松鼠症”用户，占 70%：只存不练，说明错题本更像仓库，用户没有被有效引导复习。
- “刷题型”用户，占 20%：存了就练，但无针对性，说明用户有复习意愿，但缺乏系统指导，不知道该优先练什么。
- “精英型”用户，占 10%：能够自我闭环，这类用户自驱力强，不是 MVP 的优先服务对象。

所以本次优先服务的人群是：

> MVP 优先服务“松鼠症”和“刷题型”学生，即已经录入错题但缺少复习动力或复习方向的用户；暂不优先服务能自我闭环的精英型用户。

### 3.3 核心瓶颈

当前错题本的核心瓶颈在于“有数据，但没有形成诊断能力”。

首先，学生缺少归纳能力。面对大量错题，普通学生很难从单题中抽象出共性问题，只能看到“这道题错了”，但看不到“我最近在函数概念、审题方式或解题步骤上反复出错”。因此，错题数量越多，学生越容易陷入信息过载。

其次，现有推送算法偏旧。虽然系统已经建立知识图谱，但推荐逻辑仍然停留在推原题或随机推相似题，缺少对学生掌握度的动态评估。结果是复习内容不一定命中真正薄弱点，学生也难以感知复习价值。

因此，MVP 需要先补齐两个能力：

> AI 错因归纳能力，以及基于掌握度的复习优先级排序能力。

### 3.4 学生使用习惯

从学生使用习惯看，错题本的核心设计约束是：晚间高峰、拍照为主、低耐心、低复练。

错题录入高峰集中在 21:00-22:30，占比 55%，多发生在当天作业订正后。这个时间段学生通常已经完成较长时间学习，精力和耐心有限，因此产品不应要求学生进行复杂编辑或长流程操作。

约 60% 的错题通过拍照自动录入，说明拍照是错题录入的主入口。但其中 25% 存在 OCR 识别偏差，尤其是公式和图形类题目。这意味着 AI 归因链路不能完全依赖识别结果，必须提供关键字段确认和人工修正能力，避免错误识别影响后续知识点归类和复习推荐。

学生平均每次打开错题本停留不到 2 分钟，且 70% 仅浏览不练习，说明当前错题本没有有效把“看错题”转化为“练错题”。

所以本次交互设计会尽量遵循：

> 少填写、快确认、自动归因、直接复练。

## 4. 产品目标

### 4.1 业务目标

1. 错题“错因归纳”自动化覆盖率达到 90%。
2. 二次练习率从 5% 提升到 15%。

### 4.2 产品目标

围绕业务目标，产品需要完成两件事：

1. 为了提升错因归纳自动化覆盖率：提供 AI 自动错因归纳、知识点标签和用户确认机制。
2. 为了提升二次练习率：错题卡生成后，直接引导学生完成一次低门槛复练。

## 5. MVP 范围

### 5.1 MVP 验证命题

本次 MVP 只验证：

> 学生录入一道错题后，AI 能否帮助他理解错因，并生成可复习、可复测的错题卡。

### 5.2 MVP 包含

本次 MVP 只包含 3 个核心功能：

1. 错题录入
   - 手动输入题干
   - 填写自己的错误答案
   - 拍照上传只做入口展示，不做真实 OCR

2. AI 错因分析
   - 输出知识点
   - 输出错因类型
   - 输出一句错因解释
   - 输出复习建议
   - 支持用户确认或编辑

3. 复习队列与复练反馈
   - 自动把错题加入今日复习
   - 用户点击开始复练
   - 标记“已掌握”或“仍不会”


## 6. 核心流程

主流程：

> 输入一道错题 -> AI 给出错因分析 -> 生成错题卡 -> 加入今日复习 -> 标记掌握状态

关键设计原则：

- 录入要轻：MVP 以手动输入为主，拍照只作为入口展示。
- 分析要清楚：AI 只输出学生最需要知道的知识点、错因和复习建议。
- 复练要直接：错题卡生成后直接进入今日复习，减少用户思考成本。
- 反馈要闭环：学生完成复练后标记“已掌握 / 仍不会”。

## 7. 功能设计

### 7.1 错题录入

用户动作：

- 手动输入题干。
- 填写自己的错误答案。
- 可点击拍照上传入口，但 MVP 阶段不做真实 OCR 识别。

系统动作：

- 保存题干和错误答案。
- 进入 AI 分析状态。

产出结果：

- 一条待分析的错题记录。

### 7.2 自动错因分析

用户动作：

- 查看 AI 分析结果。
- 确认或修改知识点、错因类型。

系统动作：

- 提取知识点标签。
- 判断错因类型。
- 生成一句错因解释。
- 给出一条复习建议。

错因类型初版包括：

- 概念不清
- 粗心计算
- 审题错误
- 方法不熟
- 步骤缺失

产出结果：

- 一张结构化错题卡。

### 7.3 复习队列与复练反馈

用户动作：

- 查看今日复习中的错题卡。
- 点击开始复练。
- 标记“已掌握”或“仍不会”。

系统动作：

- 错题卡生成后自动加入今日复习。
- 根据用户反馈更新掌握状态。

产出结果：

- 形成“错题录入 -> 归因 -> 复习 -> 掌握反馈”的闭环。

## 8. 系统能力设计

AI 在 MVP 中只承担三类任务：

### 8.1 知识点提取

从题干和学生错误答案中提取一个主要知识点。

### 8.2 错因归纳

根据题目、正确思路和学生错误答案，判断主要错因。

输出格式示例：

```json
{
  "knowledgePoint": "函数单调性",
  "mistakeType": "概念不清",
  "reason": "学生将函数值变化和自变量变化关系混淆，未正确判断单调区间。"
}
```

### 8.3 复习建议

根据错因生成具体复习建议，例如：

- 先复习对应概念
- 再完成 1 道同知识点基础题
- 最后完成 1 道变式题

为了控制复杂度，MVP 不做多轮推理、复杂知识图谱推荐和真实 OCR 纠错，只展示一次 错因归纳结果，并支持用户确认或编辑。

## 9. 页面设计

### 9.1 错题录入页

展示：

- 拍照上传入口，仅作为 UI 展示
- 手动输入题干入口
- 学生错误答案输入框
- 提交分析按钮

### 9.2 AI 分析与错题卡页

展示：

- 题目内容
- 知识点标签
- 错因类型
- 一句自动错因解释
- 一条复习建议

### 9.3 今日复习与反馈页

展示：

- 今日复习队列

- 反馈后的状态变化

## 10. 项目技术实现说明

本项目作为面试演示 MVP，采用纯前端实现，不拆分前后端，不接入数据库。

推荐技术栈：

- Vue 3

## 11. 数据结构

MVP 可使用前端 Mock AI 分析函数，不依赖真实后端。

核心数据结构约定如下。由于项目使用 Vue + JavaScript，不强制使用 TypeScript，可用普通 JS 对象或 JSDoc 说明字段。

```js
const mistakeCard = {
  id: "m_001",
  questionText: "已知函数 f(x)=x^2-2x，求单调区间",
  studentAnswer: "全体实数上递增",
  knowledgePoint: "二次函数单调性",
  mistakeType: "concept",
  mistakeTypeLabel: "概念不清",
  aiReason: "学生没有结合抛物线开口和对称轴判断单调区间，误以为函数始终递增。",
  reviewSuggestion: "先复习二次函数对称轴 x = -b/2a，再完成 1 道同类题。",
  recommendationReason: "该题还没有完成二次练习，建议今天优先复练，避免错题只记录不消化。",
  masteryStatus: "unreviewed",
  createdAt: "2026-06-14T21:10:00.000Z"
}
```

字段约定：

- `mistakeType` 可选值：`concept`、`careless`、`reading`、`method`、`steps`。
- `masteryStatus` 可选值：`unreviewed`、`needs_practice`、`mastered`。
- `recommendationReason` 用于说明该题进入复习队列或被优先展示的原因，MVP 阶段基于掌握状态生成。

## 12. 接口与数据约定

MVP 阶段使用前端 Mock AI 分析函数，不依赖真实后端。以下接口用于说明前后端数据边界，后续可以替换为真实服务。

```json

  "questionText": "已知函数 f(x)=x^2-2x，求单调区间",
  "studentAnswer": "全体实数上递增"
}
```

响应：

```json
{
  "id": "m_001",
  "questionText": "已知函数 f(x)=x^2-2x，求单调区间",
  "studentAnswer": "全体实数上递增",
  "knowledgePoint": "二次函数单调性",
  "mistakeType": "concept",
  "mistakeTypeLabel": "概念不清",
  "aiReason": "学生没有结合抛物线开口和对称轴判断单调区间，误以为函数始终递增。",
  "reviewSuggestion": "先复习二次函数对称轴 x = -b/2a，再完成 1 道同类题。",
  "masteryStatus": "unreviewed",
  "createdAt": "2026-06-14T21:10:00.000Z"
}
```

说明：

- MVP 阶段该接口由前端 Mock 实现。
- 真实上线时，后端可在该接口中调用 LLM，返回结构化错因分析结果。

### 12.2 获取今日复习队列

```http
GET /api/reviews/today
```

响应：

```json
{
  "items": [
    {
      "id": "m_001",
      "questionText": "已知函数 f(x)=x^2-2x，求单调区间",
```

说明：

- MVP 阶段今日复习队列由前端本地状态生成。

### 12.3 更新复练结果

```http

```


响应：

```json

  "masteryStatus": "mastered"

```

说明：

- `masteryStatus` 可选值为 `unreviewed`、`needs_practice`、`mastered`。
- MVP 阶段只更新前端本地状态。

## 13. 指标与验证

### 13.1 核心指标

- 错因归纳自动化覆盖率：目标 90%
- 二次练习率：从 5% 提升到 15%

### 13.2 辅助指标

- 错题卡生成完成率
- AI 分析结果编辑率
- 今日复习任务完成率
- 单次录入到生成卡片耗时

### 13.3 MVP 验证方式

由于本次为面试 Demo，MVP 可以通过模拟数据验证流程是否成立：

- 用户能否在 3 步内完成错题入库
- AI 是否能输出结构化错因
- 错题卡是否能进入复习队列
- 用户是否能完成一次二次练习并更新掌握状态

## 14. 风险与应对

### 14.1 OCR 识别错误

风险：

公式、图形类题目容易出现识别偏差，导致后续自动归因错误。

应对：

MVP 阶段不做真实 OCR，仅保留拍照入口和手动输入能力。后续版本再接入 OCR，并提供识别结果确认。

### 14.2 自动错因判断不准

风险：

AI 可能将“概念不清”误判为“粗心计算”。

应对：

允许用户修改错因类型。后续可根据用户修改数据优化分类策略。

### 14.3 学生只浏览不复练

风险：

70% 学生只浏览错题，不主动二次练习。

应对：

错题卡生成后直接进入今日复习队列，用“今日 3 题”“1 分钟复测”等低门槛任务推动复练。

### 14.4 流程过长导致放弃

风险：

学生晚间使用时间短，如果流程复杂，会降低完成率。

应对：

MVP 控制在“录入 -> 确认 -> 生成卡片”三步内完成错题入库。
